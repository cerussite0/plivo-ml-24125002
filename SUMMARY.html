Architecture, optimizer, schedule ablations

Best Configuration (Run 5)

  • BPE Tokenizer (vocab 1024 / 768 merges) — Trained on raw corpus bytes. Compresses Hindi Devanagari text at roughly 5.0x, removing the byte-level token inflation that was holding back earlier runs.
  • Rotary Positional Embeddings (RoPE) — Replaces learned position embeddings, freeing 49.1K parameters. This allowed doubling the context block size to 512 tokens (~1,400 bytes) without increasing the parameter count.
  • SwiGLU MLP + RMSNorm — Gated activations with hidden dimension 512. RMSNorm used throughout instead of LayerNorm (no learnable bias, fewer parameters).
  • Tied Embedding Weights — Input embedding matrix shared with the output projection head.
  • AdamW — Cosine decay schedule (peak LR 6e-4, floor 6e-5, 100-step warmup), weight decay 0.1, gradient accumulation over 8 microbatches (effective batch size 64).

Methodology

Analysis & Decision-Making

  • Diagnosed Devanagari byte-inflation as the primary bottleneck; estimated compression headroom before building the BPE tokenizer.
  • Hypothesized that Muon's Newton-Schulz updates would underperform AdamW at the 2M-parameter scale due to insufficient matrix rank — confirmed experimentally in Runs 7a/7b.
  • Traced label smoothing and QK-Norm regressions to their effect on raw NLL: both redistribute probability mass away from the correct token, which directly inflates BPB.

Implementation

  • Built a numpy-vectorized BPE training loop (merge-pair counting + fast corpus re-tokenization).
  • Implemented RoPE, SwiGLU, RMSNorm, and the Muon optimizer (5th-order Newton-Schulz iteration) in pure PyTorch.
  • Ran all experiments sequentially under identical evaluation harness to keep results comparable.
